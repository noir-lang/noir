This page describes how to place and structure Noir documentation so it stays useful for readers, maintainers, search, and AI tools.

## Choose the Right Section

Use the section that matches the reader's intent:

- **Getting Started**: first-time setup, first project, and basic project anatomy.
- **Tutorials**: guided end-to-end learning experiences that build something complete.
- **How-To Guides**: task-oriented workflows for readers who already know the basics.
- **Explainers**: conceptual background, tradeoffs, and mental models.
- **The Noir Language**: language syntax, semantics, modules, packages, crates, and standard library narrative docs.
- **Reference**: exact command, API, configuration, generated output, and behavior details.
- **Tooling**: editor support, debugging, testing, profiling, fuzzing, and development environments.

When a page seems to fit multiple sections, prefer the primary reader question. For example, "How do I use oracles?" belongs in How-To Guides, while "What are oracles and why are they unsafe unless constrained?" belongs in Explainers or Concepts.

## Page Shape

A durable docs page should have:

- frontmatter with a clear `title` and `description`
- a short opening paragraph that says what the page is for
- prerequisites when the page assumes prior context
- headings that form a skimmable outline
- runnable or source-backed examples where possible
- related links to prerequisites, follow-up pages, and reference material

Avoid starting a page with implementation detail before explaining the page's scope. Avoid duplicating long explanations that already exist elsewhere; link to the canonical page instead.

## Generated and Versioned Docs

Do not manually edit generated outputs such as `docs/processed-docs` or `docs/processed-docs-cache`. Current authored docs live under `docs/docs`.

Do not edit `docs/versioned_docs` or `docs/versioned_sidebars` unless the change intentionally targets a released docs version. Current unreleased docs are cut into versioned snapshots during release.

Generated reference material should identify its source. For example:

- Nargo command reference is generated from the `nargo` binary during the docs build.
- NoirJS and Noir WASM reference pages are generated by TypeDoc.
- Standard library API docs are generated by `nargo doc` and published separately.

## Contributor Checklist

Before opening a docs PR, check that:

- the page is in the section that matches the reader's intent
- frontmatter has `title` and `description`
- the opening paragraph gives the page context without requiring sidebar knowledge
- new pages have a sensible `sidebar_position` when order matters
- examples link to or include source-backed code where practical
- related pages are linked directly
- generated docs and versioned docs are not edited accidentally
- `yarn build` succeeds from the `docs` directory when the change affects the Docusaurus site
