1//! Inherent type implementations collection and method declaration.
2//!
3//! This module handles the collection phase of impl blocks, where methods are declared
4//! and registered in their appropriate modules. This is distinct from elaboration, where
5//! method bodies are resolved and type-checked (which happens later through regular function
6//! elaboration).
7//!
8//! ## Design
9//!
10//! The impl collection process occurs in several phases:
11//!
12//! 1. Function metadata definition
13//!    - Resolves the impl's self type (e.g., `impl Foo` → resolves `Foo` to a concrete type)
14//!    - Collects function signatures for all methods in the impl block
15//!    - This happens early so method signatures are available before bodies are elaborated
16//!
17//! 2. Method declaration
18//!    - Declares methods in the struct's module (not the impl's module)
19//!    - Handles method shadowing (inherent impls take precedence over trait impls)
20//!    - Validates that impls are only defined for types in the current crate
21//!    - This phase makes methods discoverable via `TypeName::method` syntax
22//!
23//! 3. Method elaboration
24//!    - Elaborates method bodies using regular function elaboration
25//!    - Type-checks method implementations
26//!    - This is straightforward since methods are just functions with an implicit self parameter
27//!
28//! ## Cross Module Resolution Strategy
29//!
30//! Impl methods can be declared in one module but resolved in another.
31//!
32//! - Declaration module: The struct's defining module (where the type was defined)
33//!   - Methods are added here so `TypeName::method` resolves correctly
34//!   - This allows qualified method calls from anywhere with the right imports
35//! - Resolution module: The impl block's module (where the impl appears)
36//!   - Names inside method bodies resolve in this scope
37//!   - This determines what imports, types, and functions are visible to the method
38//!
39//! ### Example:
40//! ```noir
41//! // In module `types`:
42//! struct Point { x: Field, y: Field }
43//!
44//! // In module `methods`:
45//! use crate::types::Point;
46//! impl Point {
47//!     fn distance(self) -> Field {
48//!         // This resolves names in `methods` module
49//!         helper_function() // looks for helper_function in `methods`
50//!     }
51//! }
52//! ```
53//!
54//! - `distance` is declared in the `types` module (with `Point`)
55//! - `distance` body resolves names in the `methods` module
56//! - Users can call `Point::distance` if they import `Point`
57//!
58//! ## Method Shadowing and Specialization
59//!
60//! When multiple methods with the same name are declared:
61//!
62//! 1. Inherent impls shadow trait impls: If both an inherent impl and a trait impl
63//!    define the same method, the inherent impl version takes precedence for qualified
64//!    calls like `Foo::method()`. The trait impl version is removed from the module scope
65//!    to prevent ambiguity.
66//! 2. Specialization: Multiple trait impls can define the same method if their
67//!    self types don't overlap. The method is removed from module scope when a duplicate
68//!    is found, preventing qualified access. However, the methods are still registered
69//!    in the interner for dynamic dispatch. Overlap checking happens later during
70//!    trait resolution.
71//!
72//! ## Restrictions
73//!
74//! - Foreign impl check: Inherent impls are only allowed on types defined in the
75//!   current crate. This prevents external crates from adding methods to your types.
76//! - Primitive type impls: Only the standard library can impl methods on primitive
77//!   types like `Field`, `bool`, `u32`, etc. User code cannot add methods to primitives.
78

93impl Elaborator<'_> {
94    /// Collects all impl blocks for a given type, declaring their methods in the type's module.
95    ///
96    /// This is called during the impl collection phase (after function metadata has been defined).
97    /// It validates impl generics and declares each method so they can be resolved via
98    /// `TypeName::method` syntax.
99    ///
100    /// # Parameters
101    /// - `module`: The module where the impl block appears (used for name resolution in method bodies)
102    /// - `impls`: All impl blocks for this self type
103    /// - `self_type`: The type being implemented (e.g., `Foo` in `impl Foo { ... }`)
104    ///
105    /// # Panics
106    /// If the self_type is not already resolved in each impl's function set.
107    /// The self type should be resolved by [Self::define_function_metas] before this method is called.
108    pub(super) fn collect_impls(
109        &mut self,
110        module: LocalModuleId,
111        impls: &mut [(UnresolvedGenerics, Location, UnresolvedFunctions)],
112        self_type: &UnresolvedType,
113    ) {
114        self.local_module = Some(module);
115
116        for (generics, location, unresolved) in impls {
117            self.check_generics_appear_in_types(generics, &[self_type], &[]);
118
119            self.recover_generics(|this| {
120                let no_trait_id = None;
121                this.declare_methods_on_data_type(no_trait_id, unresolved, *location);
122            });
123        }
124    }
125
126    /// Declares methods in the appropriate module and registers them in the interner.
127    ///
128    /// This handles the cross-module strategy: methods are declared in the data type's module
129    /// (for qualified calls) but will be resolved in the impl's module (for name resolution).
130    ///
131    /// # Parameters
132    /// - `trait_id`: `Some(trait_id)` if this is a trait impl, `None` for inherent impls
133    /// - `functions`: The functions/methods to declare (self_type must already be resolved)
134    /// - `location`: Location of the impl block for error reporting
135    ///
136    /// # Error Cases
137    /// - Foreign impl: Attempting to impl a type from another crate
138    /// - Duplicate methods: Multiple methods with the same name (handled via shadowing)
139    /// - Primitive impl: Non-stdlib code trying to impl methods on primitive types
140    ///
141    /// # Panics
142    /// If the self_type is not already resolved in each impl's function set.
143    /// The self type should be resolved by [Self::define_function_metas] before this method is called.
144    pub(super) fn declare_methods_on_data_type(
145        &mut self,
146        trait_id: Option<TraitId>,
147        functions: &UnresolvedFunctions,
148        location: Location,
149    ) {
150        let self_type = functions.self_type.as_ref();
151        let self_type = self_type
152            .expect("Expected data type's self type to be set before declare_methods_on_data_type");
153
154        let function_ids = functions.function_ids();
155
156        if let Type::DataType(data_type, _) = &self_type.follow_bindings() {
157            let data_ref = data_type.borrow();
158
159            // `impl`s are only allowed on types defined within the current crate
160            if trait_id.is_none() && data_ref.id.krate() != self.crate_id {
161                let type_name = data_ref.name.to_string();
162                self.push_err(DefCollectorErrorKind::ForeignImpl { location, type_name });
163                return;
164            }
165
166            // Grab the module defined by the data type. Note that impls are a case
167            // where the module the methods are added to is not the same as the module
168            // they are resolved in.
169            let module = Self::get_module_mut(self.def_maps, data_ref.id.module_id());
170
171            // Declare each method in the data type's module for qualified access (TypeName::method)
172            for (_, method_id, method) in &functions.functions {
173                let name = method.name_ident().clone();
174                let result = if let Some(trait_id) = trait_id {
175                    module.declare_trait_function(name, *method_id, trait_id)
176                } else {
177                    module.declare_function(name, method.def.visibility, *method_id)
178                };
179
180                // Handle method shadowing when a duplicate method name is found
181                if result.is_err() {
182                    let existing = module.find_func_with_name(method.name_ident()).expect(
183                        "declare_function should only error if there is an existing function",
184                    );
185
186                    // Inherent impls take precedence over trait impls for qualified calls.
187                    // If the existing method is from a trait impl, remove it from module scope
188                    // so that `TypeName::method` resolves to the inherent impl version.
189                    //
190                    // For trait-impl vs trait-impl duplicates, we also remove the existing
191                    // method to prevent qualified access. This allows specialization (e.g.,
192                    // `impl Trait<A> for Foo` and `impl Trait<B> for Foo` can coexist).
193                    // Checking whether the object types in each method overlap (which will be rejected)
194                    // happens later during trait resolution.
195                    if self.interner.function_meta(&existing).trait_impl.is_some() {
196                        module.remove_function(method.name_ident());
197                    }
198                }
199            }
200
201            // Trait impl methods are already declared in NodeInterner::add_trait_implementation
202            if trait_id.is_none() {
203                self.declare_methods(self_type, &function_ids);
204            }
205        // We can define methods on primitive types only if we're in the stdlib
206        } else if trait_id.is_none() && *self_type != Type::Error {
207            if self.crate_id.is_stdlib() {
208                // Trait impl methods are already declared in NodeInterner::add_trait_implementation
209                if trait_id.is_none() {
210                    self.declare_methods(self_type, &function_ids);
211                }
212            } else {
213                let is_primitive = self_type.is_primitive();
214                self.push_err(DefCollectorErrorKind::NonEnumNonStructTypeInImpl {
215                    location,
216                    is_primitive,
217                });
218            }
219        }
220    }
221
222    /// Registers methods in the interner's method table for dynamic dispatch.
223    ///
224    /// This associates each method with its self type, enabling method call resolution.
225    /// Unlike module declaration (which enables `TypeName::method` syntax), this registration
226    /// is required for method calls via the dot syntax (e.g., `value.method()`).
227    ///
228    /// Trait impl methods are registered separately in `NodeInterner::add_trait_implementation`,
229    /// so this is only called for inherent impls.
230    ///
231    /// # Returns
232    /// Errors if a method with the same name is already registered for this type in the interner.
233    /// This indicates a true duplicate (not specialization), such as:
234    /// ```noir
235    /// impl Foo { fn bar(self) {} }
236    /// impl Foo { fn bar(self) {} }  // Error: duplicate definition
237    /// ```
238    fn declare_methods(&mut self, self_type: &Type, function_ids: &[FuncId]) {
239        for method_id in function_ids {
240            let method_name = self.interner.function_name(method_id).to_owned();
241
242            if let Err(error) =
243                self.interner.add_method(self_type, method_name.clone(), *method_id, None)
244            {
245                self.push_err(error);
246            }
247        }
248    }
249}
